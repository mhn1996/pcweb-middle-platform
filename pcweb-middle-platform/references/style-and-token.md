# 样式与 Token

用于布局、主题、视觉一致性、CSS、CSS Modules、AntD 定制和设计清理。

## 优先级

1. AntD 组件默认值和主题 token。
2. `src/config/pageConfig.json`。
3. 现有 CSS 变量或主题机制。
4. 现有 CSS Modules。
5. 只有没有现有选项适配时，才新增语义 token 或局部样式。

## 样式规则

- 业务页面不应创建第二套视觉系统。
- 优先使用 AntD 布局原语：`Layout`、`Flex`、`Space`、grid patterns，以及适当场景下的 `Card`。
- 避免在业务页面中大量使用纯 Tailwind 实现。
- 避免过多 inline styles。
- 避免硬编码视觉值蔓延：大量一次性 hex 色值、任意间距、圆角、字号和阴影。
- 使用稳定的间距节奏，通常为 4 的倍数。
- 保持文本层级稳定：页面标题、区块标题、正文、次要文本。
- 牢记企业级 PC 的信息密度和可扫读性。

## AntD 定制

- 使用 `ConfigProvider` 和现有配置机制进行全局主题变更。
- 优先使用 token/config 覆盖，而不是直接修改源码或使用脆弱的 CSS 选择器。
- 包装 AntD 组件时，尽量保留原生 props。
- 不要在同一业务区域混用多套不兼容的按钮、输入框、弹窗或表格模式。

## CSS 指南

- 如果项目已使用 CSS Modules，就继续使用。
- 保持选择器浅层。
- 避免全局样式泄漏。
- 避免设计工具生成的类名。
- 避免导出设计代码中冗长的绝对定位链。
- 使用稳定布局约束，避免控件因数据或标签变化而跳动。

## 响应式指南

- PC Web 以桌面优先。
- 需要时使用断点重组结构。
- 不要只靠整体缩放解决响应式。
- 确保筛选、表格、弹窗和操作栏在常见笔记本宽度下仍可用。

## Token 决策树

添加值之前：

1. AntD 是否已经暴露匹配的 token？
2. `pageConfig.json` 是否已包含该值？
3. 是否有现有 CSS 变量或 class？
4. 一个语义化局部常量是否能服务多处使用？
5. 只有在以上都不满足时，才新增一次性样式。
